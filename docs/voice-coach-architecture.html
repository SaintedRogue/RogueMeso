RogueMeso · Architecture Brief

Voice Workout Coach

A hands-free, conversational assistant you talk to mid-session — log sets by voice and ask questions like "what should I be lifting on this?" or "what's my best set on incline press?" — built on a hybrid stack: local speech, a cheap cloud brain, and your existing data layer as the agent's tools.

Overview & decisions

Why this exists and the choices that shape everything downstream.

Today RogueMeso has no AI, voice, or streaming infrastructure — only Web Push. This is greenfield. But the platform already exposes the seams a voice agent needs: clean server actions for set logging, an analytics layer for e1RM / volume / PRs, and a userId-scoped data layer. The agent is mostly a thin LLM loop that maps spoken intent onto functions that already exist.

🎙️ Hybrid brain

Audio stays local & private (Whisper + Piper, free). The tool-calling brain is a cheap cloud LLM — reliable tool-calling is the real bottleneck, and a $0.10/M model nails it where a CPU-hosted small model would flake. You spend cloud money only where it buys reliability.

🪜 Phased rollout

Phase 1: tap-to-talk inside the existing app — no new framework, proves the agent + tool design. Phase 2: continuous / barge-in via a Pipecat container that reuses the exact same tool layer. Phase 1 de-risks the expensive part.

🖥️ Self-hosted host

Runs on the Unraid box (CPU). faster-whisper base/small handles short turns fine on CPU; Piper is CPU-native. The 4090 box is an escape hatch — repoint STT_URL to run large-v3 later. Not required for v1.

🧩 One tool layer, two callers

The substance is the tool layer, written once in TypeScript against existing logic. The Phase-1 route handler calls it in-process; the Phase-2 Pipecat agent calls it over HTTP. Single source of truth for auth, ownership, and business rules.

Hard prerequisite The microphone (getUserMedia) only works in a secure context — https:// or localhost. Your app serves LAN-only over http://10.0.0.232:3000, so the mic is blocked on your phone until a TLS reverse proxy (SWAG / Caddy) fronts it. This is the #1 blocker — resolve before any voice code, and ensure the proxy passes streamed responses through without buffering.

Why not fully-local? You have a 4090, but on alternate hardware you'd rather not commit. The unglamorous truth: a CPU-hosted small model is unreliable at tool-calling specifically — the one thing this feature lives or dies on. Renting a $0.10/M cloud model for the brain (cents per session) while keeping the bandwidth-heavy, privacy-sensitive audio local is the best reliability-per-dollar. If you later dedicate the 4090, swapping llm.ts to a local Qwen3-14B is a one-file change.

The tool layer — the real product

Not the voice plumbing — this is where the engineering value lives, and it's identical in both phases.

Your existing functions already encode the hard parts: ownership checks, e1RM math, the set lifecycle. The agent maps spoken intent → these functions. The key refactor is extracting the logic out of the RSC-only "use server" actions so it's callable from a route handler (or HTTP) without redirect() baggage.

The one correction worth internalizing Don't call "use server" actions from the route handler. They embed requireUser() → redirect("/login") and revalidatePath semantics meant for page renders — wrong in a fetch/stream context. Extract userId-parameterized functions (logSetFor) into setOps.ts, keep assertSetOwner as the security boundary, and make the existing actions.ts functions thin wrappers. Existing workout-UI callers stay unchanged. 
// src/lib/coach/setOps.ts — logic extracted; userId is a param, no redirect machinery
export async function logSetFor(userId, setId, weight, reps){
  await assertSetOwner(setId, userId);   // security boundary stays here
  const set = await prisma.exerciseSet.update({ /* … weight, reps, status … */ });
  await recomputeRollups(set.dayExerciseId);
  revalidateForDay(set);                // safe + useful: keeps PWA fresh for next nav
  return set;
}

// src/lib/actions.ts — now a thin wrapper; UI callers untouched
"use server";
export async function logSet(setId, weight, reps){
  const me = await requireUser();      // redirect lives only here, at the RSC edge
  return logSetFor(me.id, setId, weight, reps);
}

Phase 1 · Tap-to-talk MVP

No new framework. Proves the agent loop, the tools, and the iOS-PWA audio path.

01CaptureTap button → getUserMedia + MediaRecorder; POST the audio turn
02Transcribe/api/coach → STT sidecar → text
03ReasonrunCoach: cloud LLM with tool schemas, loops tool calls
04ActexecuteTool → setOps + read fns, under the user's identity
05SpeakPiper TTS → stream text + audio back over SSE; play clip on gesture

The streaming endpoint (v16-correct)

A ReadableStream in a Response — the blessed Web-Streams path, no Next wrapper needed. Auth via getCurrentUser() + manual 401 (the proven pattern from src/app/api/push/action/route.ts), not requireUser() which redirects.

export const runtime = "nodejs";        // Prisma + server libs
export const dynamic = "force-dynamic";  // reads cookie/body

export async function POST(req){
  const me = await getCurrentUser();      // no redirect in a handler
  if(!me) return new Response("…",{status:401});
  const audio = await req.arrayBuffer();
  const stream = new ReadableStream({ async start(c){
    const t = await transcribe(audio);              send("transcript",{t});
    for await(const turn of runCoach(me, t)) send(turn.type, turn);
    send("done",{}); c.close();
  }});
  return new Response(stream,{ headers:{
    "Content-Type":"text/event-stream",
    "Cache-Control":"no-cache, no-transform",
    "X-Accel-Buffering":"no"          // defeat reverse-proxy buffering
  }});
}
v16 gotchas confirmed against installed docs Do not add unstable_instant — that's a page-navigation export, irrelevant to route handlers. Middleware is renamed "Proxy" (src/proxy.ts); add /api/coach to its matcher's negative-lookahead so API callers get a real 401 instead of a 302 → /login.

Phase 2 · Continuous & barge-in

Only after v1 proves the tools. Adds true hands-free with interruption.

  • Pipecat container (Python) on br0: mic stream → VAD → STT → LLM → TTS with barge-in (interrupt mid-sentence). Reuses the same local STT/TTS models.
  • Reuses the Phase-1 tool layer over HTTP — a new src/app/api/coach/tool/route.ts exposes executeTool. No business logic duplicated in Python.
  • Auth by cookie-forward: the PWA forwards its signed openmeso_session cookie through Pipecat; the tool endpoint verifies it with the existing verifySession (src/lib/session.ts, already runtime-agnostic Web Crypto). The HMAC signature is the authorization — no spoofable userId. For headless use, mint a short-lived token via the existing signSession(uid).
  • The /coach screen gains a "continuous" toggle that connects to Pipecat instead of POSTing single turns.
Why this ordering matters Phase 2 is the expensive part (a second stack, WebRTC ops, Python). By making Phase 1 build the tool layer and prove the prompt/tool design first, Phase 2 becomes "add a transport," not "design the agent." The risky design work is already validated and reused verbatim.

Verification

How to know each layer actually works, end to end.

  • HTTPS/mic: open the installed PWA over HTTPS on the phone; confirm the mic grants (fails silently over plain HTTP — check the proxy first).
  • STT sidecar: curl a sample audio/mp4 (iOS) and audio/webm;opus (Chrome) clip to STT_URL; confirm transcripts (ffmpeg present).
  • Tool layer (vitest — the repo's test runner): unit-test logSetFor / getActiveSet / getBestSet with a seeded user; assert ownership rejection for another user's setId.
  • End-to-end: on the day screen say "log 185 for 8" → active set written (status:"complete", rollups updated), coach confirms by voice. Ask "what's my best set on this?" → reads back max-e1RM weight·reps. SSE renders transcript → text → audio progressively (proxy not buffering).
  • Cost: after a full session, check token spend on the LLM dashboard — expect cents/session, comfortably under $5/mo.
  • Graceful degradation: unset COACH_LLM_API_KEY → /coach shows "voice disabled", rest of app unaffected (mirrors VAPID-absent push).
